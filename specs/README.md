# Specs

Spec 描述计划中、提议中或刚被接受的变更。实现完成后，spec 不再是当前行为的 owner；当前行为必须进入 `docs/` 下的 owner 文档。

## 目录

| 目录 | 含义 |
|---|---|
| `proposed/` | 尚未接受的草案。 |
| `current/` | 已接受、正在实现或正在 active review 的 spec。 |
| `archive/` | 已完成、被拒绝或被取代的 spec。 |

## 什么时候写 Spec

当变更影响以下任一方面时，应写 spec：

- 用户可见行为。
- 架构边界或依赖方向。
- 数据、API、权限、资源命名契约。
- 部署、迁移、回滚路径。
- 外部系统集成。
- 多模块协作或跨团队约定。

很小的局部 bug fix 可以不写 spec，前提是失败测试和代码 diff 已经能自解释。

## 生命周期

1. 草案放 `proposed/`。
2. 接受实施后移到 `current/`。
3. 实施完成后，把当前行为写入 owner docs。
4. Owner docs 更新完成后，把 spec 移到 `archive/`。

## 写作要求

- 写清问题、目标和非目标。
- 写清当前事实和约束。
- 写清契约变化和 consumer 影响。
- 写清验证、迁移和回滚。
- 不写逐行 patch 指令。
