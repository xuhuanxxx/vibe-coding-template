# ADR

ADR 记录具有长期影响的工程决策和理由。它不是任务计划，也不是实施日志。

## 什么时候写 ADR

适合写 ADR 的情况：

- 选择或替换框架、存储、部署路径。
- 确定唯一真源归属。
- 决定安全、认证、权限模型。
- 划定关键数据模型边界。
- 拒绝一个看似合理、未来可能被重新提出的替代方案。

不适合写 ADR 的情况：

- 临时 TODO。
- 一次性实施步骤。
- Bug 修复流水账。
- 已经能由 `docs/constraints.md` 简单表达的普通规则。

## 命名

```text
docs/adr/0001-short-title.md
docs/adr/0002-short-title.md
```

编号递增，不复用。标题使用短横线英文或拼音均可，重点是稳定、可链接。

## 生命周期

- `proposed`：正在讨论。
- `accepted`：已接受并约束当前实现。
- `superseded`：被新 ADR 替代。
- `rejected`：明确不采用，但保留理由。

ADR 即使过时也通常不删除；将状态改为 `superseded`，并在正文写明替代 ADR。
